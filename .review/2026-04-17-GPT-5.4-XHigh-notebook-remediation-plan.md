# Notebook Audit and Remediation Plan

> Historical audit note. The active operational source of truth for notebook work is now `.review/NOTEBOOK-PLAN.md`. This document should be treated as the assessment that motivated that plan, not as a competing execution tracker.

**Date:** April 17, 2026
**Agent:** GPT-5.4 XHigh

Scope: assess whether the companion notebooks add pedagogical value and whether they are technically correct enough to serve as runnable companions.

Method: read all eight notebooks, probe the live Quokka endpoint, and run targeted local checks on the notebook formulas and helper assumptions.

## Executive verdict

The notebooks **do add value**, but mostly as pedagogical anchors rather than as trustworthy executable artifacts in their current form.

Right now they are best understood as a mixed set of:

- strong conceptual demos that need infrastructure repair;
- partially faithful demos that need relabelling or tighter claims; and
- a few notebooks that need substantial rewrite because the algorithm they claim to implement is not what they actually do.

## Global blocker

Before any notebook-specific work, fix the shared Quokka response handling used in all eight notebooks.

The current helper pattern assumes `run_qasm(...)` returns a plain counts dictionary. The live Quokka endpoint currently returns a wrapper object with top-level keys like `error`, `error_code`, and `result`, with shot data nested under `result`.

That means notebook code such as `sum(counts.values())`, `for outcome in sorted(results.keys())`, and `counts.items()` fails against the live service.

**Priority 0 task:** add one shared decoding convention across all notebooks:

- parse the Quokka payload shape actually returned now;
- convert it into a canonical `{bitstring: count}` dictionary before any downstream analysis; and
- verify one toy circuit end-to-end after the adapter change.

Until that lands, all notebook execution conclusions remain provisional.

## Per-notebook decisions

### `01-logistics.ipynb`

- **Decision:** Keep and repair
- **Why:** Good pedagogical shape. It does what the unit needs: brute-force MaxCut, build a `p=1` QAOA circuit, show outcomes, show a parameter landscape.
- **Current issues:** blocked by the Quokka response mismatch; next-unit link is stale after chapter renumbering.
- **First fix:** shared Quokka adapter, then end-to-end execution check.

### `02-cryptography.ipynb`

- **Decision:** Rewrite or explicitly relabel as a compiled period-finding toy
- **Why:** This notebook is not a faithful Shor implementation, which is acceptable for a toy example, but it is currently internally inconsistent even on its own terms.
- **Current issues:**
  - the prose says there is a 4-qubit input register, but the period-extraction code later treats only 3 bits as the input register;
  - the circuit measures all 4 qubits;
  - the oracle is a compiled phase gadget rather than a modular exponentiation circuit, but the notebook does not frame that sharply enough.
- **Recommended direction:** keep only if the notebook is reframed explicitly as a compiled toy period-finding demonstration for `N = 15`; otherwise rewrite more honestly around the compiled-oracle simplification.

### `03-drug-discovery.ipynb`

- **Decision:** Relabel and repair
- **Why:** The notebook has genuine teaching value as a VQE structure demo, but it currently overclaims what it computes.
- **Current issues:**
  - it claims to work at various bond lengths, but the bond-length argument is ignored and the Hamiltonian coefficients are constant;
  - it claims to build a potential energy surface but does not do so;
  - it inserts the `XX + YY` contribution analytically instead of measuring those terms with basis-rotation circuits.
- **Recommended direction:** either implement the missing pieces properly or relabel it as a single-geometry VQE anatomy demo for H2.

### `04-machine-learning.ipynb`

- **Decision:** Keep and repair
- **Why:** This is one of the strongest notebooks conceptually. It matches the unit's worked example, uses a concrete dataset, and the quantum kernel + classical SVM division is pedagogically clean.
- **Current issues:** mainly the shared Quokka response mismatch and stale next-unit link.
- **First fix:** shared Quokka adapter, then performance sanity check on the kernel matrix and SVM workflow.

### `05-finance.ipynb`

- **Decision:** Rewrite unless explicitly relabelled as a Grover-style toy
- **Why:** The title and framing say amplitude estimation, but the actual notebook is mostly classical Monte Carlo plus a simplified Grover-style marking of profitable bins.
- **Current issues:**
  - it does not implement amplitude estimation;
  - it does not encode payoff amplitudes in the way the manuscript now correctly describes;
  - it demonstrates marked-state amplification, not option pricing by QAE.
- **Recommended direction:** either rewrite around a real toy amplitude-estimation construction, or relabel the notebook as a Grover-based intuition builder and stop claiming that it prices the option on Quokka.

### `06-supply-chains.ipynb`

- **Decision:** Keep, but repair the math and narrow the claim
- **Why:** The notebook still helps explain the QUBO-to-Ising idea, which is the point of the unit.
- **Current issues:**
  - it advertises a 4-nurse / 4-shift problem, then collapses to a much smaller 2-nurse / 2-shift toy;
  - the current QUBO-to-Ising coefficient mapping appears wrong for the defined Q matrix, so the QAOA circuit is not faithfully encoding the toy cost function.
- **Recommended direction:** keep the small toy if you want a readable notebook, but rewrite the introduction so the scale reduction is explicit and fix the coefficient mapping before treating results as meaningful.

### `07-materials-science.ipynb`

- **Decision:** Rewrite
- **Why:** This notebook currently makes a core quantitative claim that does not survive its own formulas.
- **Current issues:**
  - the QPE section wraps a negative energy phase into `[0,1)` and then reconstructs a positive energy from the measured integer;
  - using the notebook's own formula gives an energy estimate wildly inconsistent with the exact ground-state energy;
  - the notebook also talks about observing a Mott insulator transition, which is too strong for the 2-site toy.
- **Recommended direction:** rebuild this as either a clearly labelled phase-estimation toy with an intentionally encoded positive phase, or as a classical-exact-diagonalisation notebook plus a separate QPE intuition cell that does not pretend to recover the physical energy directly.

### `08-climate-energy.ipynb`

- **Decision:** Relabel and repair
- **Why:** As a capstone pipeline notebook it is useful, but the quantum part is still largely inherited from the H2 notebook's simplifications.
- **Current issues:**
  - it uses the same analytical `XX + YY` shortcut as Unit 3 rather than measuring those terms;
  - the precomputed active-space Hamiltonian makes sense as a teaching device, but the notebook should more clearly distinguish toy active-space benchmark from a runnable catalyst-screening workflow.
- **Recommended direction:** keep the capstone structure, but label it as a pipeline demonstration with precomputed embedded Hamiltonian data unless the missing measurement steps are implemented.

## Value judgment

If the standard is "does each notebook help the reader see the shape of the algorithmic story?" then the notebook program is worth keeping.

If the standard is "are these currently correct runnable companions to the manuscript?" then the answer is no, not yet.

The best near-term strategy is not to delete them. It is to:

1. repair the shared infrastructure;
2. relabel any notebook whose current title overpromises; and
3. reserve substantial rewrites for the three notebooks whose core algorithmic claims are currently the least faithful.
